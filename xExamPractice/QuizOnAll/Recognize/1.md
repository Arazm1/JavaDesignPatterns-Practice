Recognizing the Patterns – Harjoitus 1: Tunnista malli

Seuraavat kuvaukset vastaavat tiettyjä suunnittelumalleja. Kirjoita kunkin kohdalle oikea malli (esim. Factory Method, Singleton, Strategy…).


Ohjelmassa on yksi keskusolio, joka hallitsee resurssien jakamisen koko sovelluksessa, eikä sen instanssia voi luoda useampaa.
# Singleton

Luokissa on yhtenäinen rajapinta, mutta konkreettiset luokat voivat koostua useista alielementeistä.
# Composite

Sovelluksessa on erilaisia algoritmeja, jotka voidaan vaihtaa lennossa ilman, että asiakkaan koodi muuttuu.
# Strategy

Kun objekti muuttuu, kaikki siihen riippuvaiset objektit saavat automaattisesti ilmoituksen ja päivittävät tilansa.
# Observer

Halutaan luoda objektit eri aliluokista ilman, että asiakkaan koodi tietää tarkalleen, mitä luokkaa instansioidaan.
# Factory Method

3. Recognizing the Patterns – Harjoitus 3: Koodipohjainen tehtävä

Kirjoita pseudokoodi tai lyhyt Python-/Java-esimerkki, joka havainnollistaa seuraavia:

Singleton – Luokan, joka sallii vain yhden instanssin ja tarjoaa globaali pääsyn sen instanssiin.

Decorator – Luokan, joka lisää uusia toimintoja olemassa olevaan olioon ilman, että sen koodia muutetaan.

Template Method – Abstrakti luokka, joka määrittelee algoritmin rungon, mutta jättää osan vaiheista aliluokille.


# Singleton
```java
public class Logger{
    private static Logger instance;

    private Logger(){
    }

    public static synchronized void getInstance(){
        if(instance == null){
            instance = new Logger();
        }

        return instance;
    }

    public synchronized void log(String logMessage){
        System.out.println("Logged: " + logMessage);
    }

}

public class Main {

    public static void main(String{} args){
        Logger logger1 = Logger.getInstance();
        Logger logger2 = Logger.getInstance();

        logger1.log("Hi");
        logger2.log("Hi too");

    }
}



```

4. Recognizing the Patterns – Harjoitus 4: Refaktorointitehtävä

Seuraava koodi on yksinkertaistettu versio verkkopankkisovelluksesta:

class PaymentProcessor {
    void processCreditCardPayment(...) { ... }
    void processPayPalPayment(...) { ... }
    void processCryptoPayment(...) { ... }
}

Kysymys:

Mikä suunnittelumalli sopisi paremmin erottamaan maksutapojen logiikat, jotta koodi noudattaisi avoin-suljettu-periaatetta?
# Strategy
